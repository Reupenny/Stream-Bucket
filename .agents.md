## Project Overview

**Stream Bucket** is a macOS application for batch processing video files into HLS streams with optional S3 upload and Cloudflare CDN integration.

### Key Features

- **VOD Processing**: Convert video files to HLS format with multiple bitrate renditions
- **Live Streaming**: Capture and encode live streams with automatic HLS output
- **S3 Integration**: Direct upload to S3-compatible storage (Backblaze B2, AWS S3, etc.)
- **Cloudflare CDN**: Stream content through Cloudflare's global CDN network

## Architecture

### Main Components

| ------------------------------ | ---------------------------------------------------- |

### Data Flow

```
User Input → ProcessorState (state) → BatchProcessor → FFmpegWrapper → S3Uploader → S3Client
```

## Development Guidelines

### Build Process

1. **Prerequisites**:

   - macOS 13.0+ (Ventura or later)
   - FFmpeg installed via Homebrew: `brew install ffmpeg`
2. **Build Command**:

   ```bash
   ./build.sh [version]
   ```
3. **Manual Compilation**:

   ```bash
   swiftc -parse-as-library \
       -target arm64-apple-macosx13.0 \
       -O \
       Sources/*.swift \
       -o "Stream Bucket.app/Contents/MacOS/Stream Bucket"
   ```

### Code Style

- **SwiftUI**: Use `@StateObject` for view models, `@EnvironmentObject` for shared state
- **Async/Await**: Use modern concurrency for file operations and network requests
- **MainActor**: State updates must occur on MainActor
- **Error Handling**: Use `LocalizedError` for user-facing error messages

### State Management

The `ProcessorState` class is the single source of truth:

```swift
@MainActor
class ProcessorState: ObservableObject {
    @Published var queue: [QueuedFile] = []
    @Published var isProcessing: Bool = false
    @Published var progress: Double = 0.0
    // ... other state properties
}
```

### S3 Integration Pattern

1. Credentials stored in macOS Keychain via `KeychainHelper`
2. S3Client handles AWS Signature V4 signing
3. Supports S3-compatible providers (Backblaze B2, AWS S3, R2, etc.)

## No band-aids. No "we'll clean it up later."

If a piece of code is in the wrong place, has the wrong shape, has confusing naming, or carries leftover assumptions — **fix it at the source, even if it means refactoring multiple files**.

You are explicitly authorized — and expected — to:

- Rename modules, files, types, and functions across the whole codebase to match the cleaner architecture.
- Move responsibilities between layers (e.g. push logic out of a handler into a repository, or out of a component into the engine) when that is correct.
- Delete dead code, unused exports, half-finished abstractions, and "just in case" parameters.
- Restructure folders if the current layout no longer reflects the architecture.

What you must not do:

- Wrap new logic around old logic to "avoid touching it."
- Add a second way of doing something because the first way is awkward — fix the first way.
- Leave TODO/FIXME notes about cleanup instead of doing the cleanup.
- Hide the wrong abstraction behind a thin adapter so callers "don't notice."
- Justify a workaround with "to keep this PR small" or "to avoid breaking other things." Other things can break in this PR. Fix them in this PR.

## Debugging Tips

1. **FFmpeg Issues**: Check `FFmpegWrapper.findFFmpeg()` returns valid path
2. **S3 Upload Failures**: Verify credentials in Keychain, check endpoint URL format
3. **Live Stream Problems**: Ensure port 1935 is not blocked, check RTMP ingest URL
4. **UI Not Updating**: Verify `@Published` properties are modified on MainActor

## Testing Considerations

- Test with various video formats (MP4, MOV, AVI)
- Verify HLS segment generation
- Test S3 upload with different providers
- Validate live streaming with actual RTMP source
